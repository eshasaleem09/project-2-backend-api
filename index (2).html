Project Overview

The Nervous System

If Project 1 was the skin, Project 2 is the life — engineering the backend API and ensuring architectural integrity.

This project moves beyond visual presentation into the application's brain — building robust API endpoints that handle the flow of data between frontend and server with absolute reliability.

By mastering the IPO model (Input → Process → Output), I built a system that controls inputs and predicts outputs, bridging the gap between user interaction and server-side processing through pure API logic.

The key architectural insight: resources are nouns, methods are verbs. RESTful naming conventions, proper status code communication, and the Gatekeeper Rule (never trust the client) form the foundation of a production-ready backend.

// DecodeLabs · Project 2 · Backend API
const express = require('express');
const app = express();
 
// GET endpoint — retrieve users
app.get('/users', (req, res) => {
  const users = getUsers();
  res.status(200).json({ users });
});
 
// POST endpoint — create user
app.post('/users', (req, res) => {
  const { name, email } = req.body;
  if (!name || !email)
    return res.status(400).json({
      error: 'Validation failed'
    });
  res.status(201).json({ created: true });
});

Anatomy of Action

HTTP Methods

Each method has a precise semantic meaning. Click to explore what each one does.

GET — Retrieval

Fetch a resource without modifying it. GET is safe and idempotent — calling it 100 times produces the same result as calling it once.

Safe Idempotent
GET /users
→ 200 OK
 
{
  "users": [
    { "id": 1, "name": "Alice" },
    { "id": 2, "name": "Bob" }
  ]
}

POST — Creation

Submit data to create a new resource. POST is unsafe and non-idempotent — each call can produce a different result (a new record each time).

Unsafe Non-idempotent
POST /users
 
Request Body:
{
  "name": "Charlie",
  "email": "charlie@example.com"
}
 
→ 201 Created
{ "id": 3, "created": true }

PUT — Update / Replace

Replace an existing resource entirely. PUT is idempotent — calling it multiple times with the same data has the same effect as calling it once.

Unsafe Idempotent
PUT /users/3
 
{
  "name": "Charlie Updated",
  "email": "new@example.com"
}
 
→ 200 OK
{ "updated": true }

DELETE — Removal

Remove a resource permanently. DELETE is idempotent — deleting the same resource twice should return 404 the second time, not an error in the system.

Unsafe Idempotent
DELETE /users/3
 
→ 204 No Content
 
# Second call:
DELETE /users/3
→ 404 Not Found

Server's Tone

HTTP Status Codes

Don't force the client to guess. Use the semantic code. Every response tells a story.

200 OK Standard success response for GET, PUT requests
201 Created Resource successfully created via POST
204 No Content Success but no body returned (DELETE)
400 Bad Request Malformed request or failed validation
401 Unauthorized Authentication required or failed
403 Forbidden Authenticated but lacks permission
404 Not Found Resource does not exist at this path
500 Server Error Unexpected failure — treat as a learning opportunity

Key Learnings

What This Project Proved

Building this milestone demonstrates the ability to bridge user interaction and server-side processing through pure API logic.

01

Build with Integrity

Every endpoint validates inputs, handles edge cases, and returns meaningful status codes. No guessing for the client.

02

Validate Everything

The Gatekeeper Rule: syntactic + semantic validation at every ingress point. Malformed data never reaches the database.

03

Communicate Clearly

Semantic HTTP codes make the server's intent unambiguous. 201, 400, 404, 500 — each tells the client exactly what happened.

04

Respect the Architecture

RESTful principles, statelessness, and gateway patterns are not optional — they are what makes a system scalable and maintainable.
